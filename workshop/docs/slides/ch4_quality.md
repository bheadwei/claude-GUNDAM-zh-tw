## Slide 2: /verify — 三道品質門

```
⚡ 操作：輸入 /verify

┌──────────────────────────────────────┐
│  🔍 全面驗證                          │
│                                      │
│  Gate 1: 型別檢查                    │
│    Python: mypy --strict ✅           │
│    TS:     tsc --noEmit  ✅           │
│                                      │
│  Gate 2: 測試                        │
│    pytest     → 12/12 passed ✅       │
│    vitest     → 8/8 passed  ✅       │
│    coverage   → 87%         ✅ (>80%) │
│                                      │
│  Gate 3: Lint                        │
│    ruff       → 0 errors    ✅       │
│    eslint     → 0 errors    ✅       │
│                                      │
│  結果: ✅ 全部通過                    │
└──────────────────────────────────────┘
```

**講師口述：** `/verify` 一次跑完三道門：型別、測試、lint。任何一道沒過都會告訴你哪裡有問題。這就是你提交 PR 前的最後檢查。

## Slide 3: /e2e — Playwright 自動操作瀏覽器

```
⚡ 操作：輸入 /e2e

e2e-validation-specialist agent 會：

1. 啟動 Playwright 瀏覽器
2. 自動執行使用者流程：

   打開首頁
     → 看到空的 Todo 列表
   新增一筆 "買牛奶"
     → 列表出現 "買牛奶"
   標記完成
     → 出現刪除線
   切換到「工作」分類
     → 列表為空（正確）
   刪除 Todo
     → 回到空列表

3. 產出截圖 + 測試報告
```

**講師口述：**（等 Playwright 跑完）看到了嗎？瀏覽器自己在操作。這不是 mock，是真的打開瀏覽器、點按鈕、驗證結果。這就是端到端測試。

## Slide 4: /check-quality — 全面品質報告

```
⚡ 操作：輸入 /check-quality

┌──────────────────────────────────────┐
│  📊 品質評估報告                      │
│                                      │
│  程式碼品質     8.5/10               │
│  測試覆蓋率     87%                  │
│  安全性         ✅ 無已知漏洞         │
│  可維護性       高                    │
│  技術債         低                    │
│                                      │
│  建議：                              │
│  • 可考慮加上 rate limiting           │
│  • 前端可加 loading 狀態              │
│                                      │
│  適合交付的 Agent：                   │
│  → deployment-expert (部署)           │
│  → doc-specialist (文檔)              │

**講師口述：** `/check-quality` 不只是打分，它還會告訴你接下來該找哪個 Agent。這就是模板的「智慧路由」——根據專案狀態推薦下一步。

```
⚡ 操作：輸入 /time-log

┌──────────────────────────────────────┐
│  ⏱️  開發時間報表                     │
│                                      │
│  2026-04-16                          │
│  ──────────────────────────          │
│  2.1 基礎架構          12 min        │
│  2.2 CRUD API           18 min       │
│  3.1 前端元件           15 min        │
│  3.2 CRUD 互動          12 min        │
│  4.1 Bug 修復            3 min        │
│  ──────────────────────────          │
│  合計                   60 min        │
│                                      │
│  → 你沒有手動記過任何時間              │
│  → 全部由 Hook 自動追蹤               │

**講師口述：** 最後看這個。我們從頭到尾沒有記過時間，但模板的 Hook 在背後自動追蹤每個任務花了多久。這對回顧和估算未來專案非常有用。
