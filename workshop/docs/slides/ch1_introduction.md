## Slide 3: VibeCoding 模板 = AI 開發的標準作業程序

```
┌─────────────────────────────────────────────┐
│          VibeCoding 模板                     │
│                                             │
│  📋 16 份文件模板    規範從需求到部署        │
│  🤖 13 個 AI Agent   各司其職、自動協作      │
│  ⚡ 17 個快捷指令    一行啟動完整流程        │
│  📏  9 條自動規則    品質底線自動守護        │
│  🔧  7 個領域技能    按需載入專業知識        │
│                                             │
│  → 複製到任何新專案，立即可用                │
└─────────────────────────────────────────────┘
```

**講師口述：** 這個模板就像一套開發的 SOP。你不需要每次都從零設定 AI，複製過去就能用。

## Slide 4: 對比 — 沒模板 vs 有模板

```
       沒有模板                    有模板
  ┌──────────────┐          ┌──────────────┐
  │ "幫我寫一個   │          │ /task-init   │
  │  Todo App"    │          │  ↓ 互動式Q&A │
  │       ↓       │          │  ↓ 產出 WBS  │
  │  AI 直接寫    │          │ /task-next   │
  │  一大段程式碼  │          │ /plan        │
  │       ↓       │          │ /tdd         │
  │  沒測試       │          │ /review-code │
  │  沒文件       │          │ /verify      │
  │  沒品質把關    │          │       ↓       │
  │       ↓       │          │ 有測試 ✅     │
  │  出問題才發現  │          │ 有文件 ✅     │
  └──────────────┘          │ 有品質門 ✅   │
                            └──────────────┘
```

**講師口述：** 左邊是大部分人用 AI 的方式，右邊是今天要教的方式。差別在於**流程**，不在於 AI 模型本身。

## Slide 5: 5 大擴展機制

```
┌─────────┐  系統事件     ┌──────────┐
│  Hook   │──自動觸發────→│ Shell 腳本 │  例：啟動時偵測模板
└─────────┘               └──────────┘

┌─────────┐  使用者輸入   ┌──────────┐
│ Command │── /xxx ─────→│ 預設 Prompt│  例：/tdd, /plan
└─────────┘               └──────────┘

┌─────────┐  語意偵測     ┌──────────┐
│  Skill  │──自動觸發────→│ 領域知識   │  例：偵測到 FastAPI 自動載入
└─────────┘               └──────────┘

┌─────────┐  任務委派     ┌──────────┐
│  Agent  │──隔離執行────→│ 專業 AI    │  例：code-quality-specialist
└─────────┘               └──────────┘

┌─────────┐  跨 Agent    ┌──────────┐
│ Context │──知識共享────→│ 報告存檔   │  例：品質報告 → 測試報告
└─────────┘               └──────────┘
```

**講師口述：** 模板有 5 種擴展方式。今天我們主要會用到 Command（快捷指令）和 Agent（專業 AI），其他三個先知道就好。

## Slide 6: 13 個 Agent — 各司其職

```
┌─────────── Opus（重量級推理）───────────┐
│  planner         功能規劃、步驟拆解      │
│  architect       架構設計、技術選型      │
│  security-auditor 安全漏洞掃描          │
└─────────────────────────────────────────┘

┌─────────── Sonnet（一般開發）───────────┐
│  code-quality    程式碼審查             │
│  tdd-guide       測試驅動開發           │
│  e2e-specialist  端到端測試             │
│  test-engineer   測試補強              │
│  refactor-cleaner 死碼清理             │
│  deployment      部署與 CI/CD          │
└─────────────────────────────────────────┘

┌─────────── Haiku（輕量快速）───────────┐
│  build-resolver  建置錯誤秒修           │
│  doc-specialist  文檔生成              │
│  template-mgr    模板管理              │
└─────────────────────────────────────────┘
```

**講師口述：** 不同難度的任務用不同等級的 AI。重的用 Opus 想清楚，一般開發用 Sonnet，簡單修錯用 Haiku 快速解決。你不需要手動選，模板會自動分配。

## Slide 7: 目錄結構（只看第一層）

```
.claude/
├── agents/       ← 13 個 Agent 定義
├── commands/     ← 17 個快捷指令
├── rules/        ← 9 條自動規則（每次對話載入）
├── skills/       ← 7 個領域技能
├── hooks/        ← 6 個系統事件腳本
├── context/      ← Agent 間的報告共享
├── guides/       ← 參考文件（不自動載入）
└── taskmaster-data/ ← WBS 任務 + 時間追蹤
```

**講師口述：** 所有設定都在 `.claude/` 目錄下。你不需要背，知道去哪找就好。今天會實際用到的是 commands 和 agents。
