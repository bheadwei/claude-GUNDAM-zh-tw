# 互動問答規則（Interactive Q&A）

所有需要向使用者收集資訊、確認選項或做決策的流程，**必須**遵守本規則。

## 核心原則

### 1. 一律使用 `AskUserQuestion` 工具

- ❌ **禁止**在純文字中列出多題（`1. X? 2. Y? 3. Z?`）等使用者一次回答
- ❌ **禁止**把選項寫在純文字訊息中（`請選擇 [1] [2] [3]`）
- ✅ **必須**使用 `AskUserQuestion` 工具，提供結構化選項

### 2. 一次一題（逐題互動）

- 每次 `AskUserQuestion` 呼叫**只問 1 題**（最多 2 題且必須高度相關）
- 收到答案後再決定下一題內容，可依答案動態調整後續問題
- 像 plan 模式那樣的逐步對話體驗

### 3. 選項設計

- 每題提供 **2-4 個預設選項**（使用者總能選 Other 自填）
- 若有推薦選項，放第一個並標註 `(Recommended)`
- 選項要互斥、明確、具體
- 為每個選項寫清楚 `description`，說明選擇後會發生什麼

### 4. 必須記錄問答歷史（批次寫入）

**所有問答必須寫入** `.claude/qa-history/` 以供追溯：

- 檔案路徑：`.claude/qa-history/YYYY-MM-DD-HHMMSS-<command-name>.md`
  - 例：`.claude/qa-history/2026-04-09-143055-task-init.md`
- 若目錄不存在，先建立

#### 寫入時機（重要：省 token）

- ❌ **不要**每問一題就寫一次（多次 Write/Edit 會浪費 token）
- ✅ **先在記憶中收集所有問答**，等流程全部結束後**一次性寫入整份檔案**
- ✅ 使用 `Write` 工具（不是 Edit），一次寫完整內容
- 例外：若流程極長（>10 題）可在中段寫一次做 checkpoint，否則一律最後才寫

#### 記錄檔格式

```markdown
# Q&A 記錄：<command-name>

**開始時間：** 2026-04-09 14:30:55
**指令：** /task-init
**專案：** <project-name>

---

## Q1: 專案名稱要用什麼？
**選項：**
- GCP-Scrapy（沿用目錄名稱）
- 自訂名稱

**答：** GCP-Scrapy
**時間：** 14:31:02

---

## Q2: 主要開發語言？
**選項：**
- Python (Recommended)
- TypeScript
- Go
- Java

**答：** Python
**時間：** 14:31:15

---

## 最終結論
<問答流程結束後寫入最終決策摘要>
```

## 適用範圍

任何斜線指令或 agent 在以下情境都必須遵守本規則：

- 收集專案資訊（`/task-init`）
- 選擇下一步動作（`/task-next`、`/hub-delegate`、`/check-quality`）
- 確認實作方向（`/plan`）
- 選擇 Agent 或技術方案
- 任何需要使用者做決定的情境

## 例外

以下情境**不需要**走 Q&A 流程，直接輸出即可：

- 純資訊顯示（`/task-status`、`/time-log`）
- 唯一選項（沒有分支、不需要使用者決定）
- 使用者在指令參數中已明確指定（例如 `/task-init my-project`）

## 為何這樣做？

- **使用者體驗**：逐題互動比「一次丟一堆問題」更容易專注回答
- **可追溯**：qa-history 讓未來 session 能回顧決策脈絡
- **結構化**：`AskUserQuestion` 提供明確選項，降低誤解
- **保留上下文**：問答紀錄是決策的一部分，不應只存在於揮發性的對話記憶裡
