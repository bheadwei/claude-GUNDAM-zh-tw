# TDD 指令

此指令呼叫 **tdd-guide** agent 強制執行 TDD，並**自動載入當前 WBS 任務的 plan 檔**（若存在）按階段推進。

**相關規範：** `.claude/rules/plan-persistence.md`

## 執行前流程（自動）

### 1. 偵測當前任務與計畫

tdd-guide agent 啟動時：

1. 讀 `.claude/taskmaster-data/.current-task`，取得當前 WBS 任務 ID（例如 `2.1`）
2. 在 `.claude/taskmaster-data/plans/` 中尋找 `<id>-*.md`
3. 若找到 → 顯示計畫摘要，詢問使用者：

   ```
   偵測到計畫：2.1-auth-middleware.md
     狀態：🔄 進行中（階段 2/4）
     下一階段：階段 2 - JWT 驗證核心

   動作？
     [Recommended] 接續執行階段 2
     從階段 1 重新開始
     忽略計畫，自由 TDD
   ```

4. 若**找不到計畫**但當前任務門檻達標（≥ 2 檔案 或 ≥ 1h 預估）→ 提示：

   ```
   當前任務 2.1 尚無計畫。建議先執行 /plan 2.1 再開始。
   仍要直接 TDD 嗎？(y/N)
   ```

5. 若**無當前任務** → 進入 ad-hoc TDD，不讀計畫檔

### 2. 載入階段

若使用者選擇接續執行：

- 讀取 plan 檔對應階段的「驗收」條件作為 TDD 的 RED 目標
- 以「驗收條件」推導測試，以階段步驟推導實作

## TDD 循環（每階段）

```
RED → GREEN → REFACTOR → 更新 plan 階段狀態
```

### 1. 先寫測試 (RED)
依階段「驗收」推導失敗測試。

### 2. 執行測試 — 確認失敗
```bash
npm test
```

### 3. 寫最小實作 (GREEN)
只寫足以通過的程式碼。

### 4. 執行測試 — 確認通過

### 5. 重構 (IMPROVE)
保持綠燈，消除重複、改善命名。

### 6. 驗證覆蓋率
```bash
npm run test:coverage
# 要求: 80%+
```

### 7. 更新 plan 階段狀態（若從 plan 執行）

- 讀 plan 檔，將該階段標 ✅
- 更新 frontmatter：`current_phase`、`updated`
- 若為最後階段 → `status: "✅ 完成"`，提示使用者執行 `/verify`
- 同步更新 `plans/INDEX.md`

## 覆蓋率要求

- **80% 最低**適用於所有程式碼
- **100% 要求**：財務計算、認證邏輯、安全關鍵程式碼、核心商業邏輯

## 使用時機

- 實作新功能
- 新增函式/元件
- 修復 bug（先寫重現 bug 的測試）
- 重構現有程式碼
- 建構關鍵商業邏輯

## 最佳實踐

**應該做：**
- 在實作之前先寫測試
- 執行測試確認失敗後再實作
- 寫最小程式碼讓測試通過
- 測試綠燈後才重構
- 加入邊界情況與錯誤場景
- 關鍵程式碼目標 100% 覆蓋率

**不應該做：**
- 先寫實作再寫測試
- 跳過每次變更後的測試執行
- 一次寫太多程式碼
- 忽略失敗的測試
- 測試實作細節（應測試行為）
- mock 所有東西（優先整合測試）

## 與其他指令的搭配

```
/plan             規劃並寫入 plans/<id>-<slug>.md
/tdd              自動載入當前 plan，按階段 TDD 推進
/build-fix        建置錯誤修復
/review-code      程式碼審查
/verify           驗證覆蓋率與整體驗收標準 → 標 WBS 為 ✅
```
